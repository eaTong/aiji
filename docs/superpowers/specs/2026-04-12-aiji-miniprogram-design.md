## 一、产品定位

**AI己** 是一款面向健身人群的微信小程序，定位为 **AI驱动的全能健身管家**。

**目标用户**：
- 入门健身者（A）：刚开始健身，需要大量引导、计划和教育
- 中级健身爱好者（B）：有一定经验，需要优化训练、精细追踪数据

**核心差异化**：
- AI全程陪伴：24小时在线，随时问随时答
- 数据整合中心：训练 + 饮食 + 身体数据全打通
- 极简记录体验：最低摩擦完成日常记录
- 个性化计划：AI根据用户情况生成方案并动态调整

## 二、交互架构

### 设计原则

**AI对话是唯一主入口**，所有功能通过对话触达，数据自动沉淀到各模块后台。

**交互模式**：混合模式
- 自由对话：用户随便说，AI理解意图并处理
- 快捷按钮：对话框上方常用操作，一键进入对应流程

### 底部导航（4 Tab）

```
[🤖 AI助手]  [📊 数据]  [💪 训练]  [👤 我]
```

|------|------|
| 📊 记体重 | 弹出快速输入表单（数字键盘） |
| 💪 记训练 | 进入训练记录对话流 |
| 🍽️ 记饮食 | 进入饮食记录对话流（含拍照） |

#### 训练推荐快捷按钮

**入口**：快捷操作栏「🎯 练什么」按钮

**触发流程**：
```
点击按钮 → 系统收集上下文 → AI生成推荐 → 展示推荐卡片 → 用户决策
```

**推荐依据（优先级排序）**：

| 优先级 | 因素 | 数据来源 |
|-------|------|---------|
| P0 | 健身目标 | 个人档案 |
| P0 | 可用器材 | 个人档案 |
| P1 | 训练历史（近7天） | 训练记录数据库 |
| P1 | 当前恢复状态 | 恢复评分/肌群热力图 |
| P1 | 今日计划状态 | 训练计划表 |

**推荐卡片展示**：
- 训练名称、预计时长、训练类型
- 动作预览（4-6个动作）
- 推荐原因（结合用户实际情况）

**交互按钮**：
| 按钮 | 行为 |
|-----|------|
| 开始训练 | 进入「今日训练」执行界面，预填推荐动作 |
| 换一个 | 重新生成推荐（最多3次） |
| 查看计划 | 跳转 Tab 3「我的计划」 |

**异常场景处理**：

| 场景 | AI 响应 |
|-----|---------|
| 今日已完成训练 | 夸赞+休息建议（拉伸/轻度有氧） |
| 无健身目标 | 引导式提问（减脂/增肌/塑形/提升体能） |
| 无器材信息 | 引导式提问（健身房/哑铃/徒手） |
| 所有肌群未恢复 | 强制休息建议 |
| 新用户无记录 | 推荐全身训练（新手友好版） |
| 网络异常 | 友好提示+备选入口 |

**与 Tab 3 训练计划的关系**：
- 独立但协同：快捷按钮提供即时单次推荐，不影响正式计划
- 数据共享：复用动作库、用户档案、训练记录、恢复状态算法
- 计划优先：用户有进行中计划时，优先展示计划内容

#### 场景化对话流

**新用户引导流（首次启动）**

AI依次通过对话收集：
1. 身高 / 体重 / 年龄 / 性别
2. 健身目标（减脂 / 增肌 / 塑形 / 提升体能）
3. 当前水平（零基础 / 有一定基础 / 较有经验）
4. 可用器材（健身房 / 哑铃 / 徒手）
5. 每周可训练天数

完成后自动生成初始训练计划。器材设置可在「我 → 个人档案」中随时修改，修改后 AI 自动适配后续训练计划。

**训练记录对话流**

AI解析用户输入后，生成**可编辑卡片**，用户可直接在卡片内修改后保存。

**卡片字段**：

| 字段 | 类型 | 编辑方式 |
|-----|------|---------|
| 动作名称 | string | 下拉选择器 |
| 组数 | number | 数字输入 |
| 次数 | number | 数字输入 |
| 重量 | number | 数字输入 |
| 热身组 | boolean | 圆形切换按钮 |

**肌肉容量记录：**

每次保存组次时，系统根据动作的**主肌群 + 辅肌群**计算并记录 `muscleVolumes`（Json 字段）：
- 主肌群：重量 × 次数 × 1.0
- 辅肌群：重量 × 次数 × 0.5

围度/体重趋势图支持叠加显示训练容量曲线。

**交互示例**：
```
用户输入：「今天练胸，杠铃卧推60公斤10次4组」
         ↓
AI展示可编辑卡片：
┌─────────────────────────────────────────────────────┐
│ 💪 训练记录                                          │
├─────────────────────────────────────────────────────┤
│ 📅 2026-04-12  胸部训练                            │
│                                                      │
│ ┌───────────────────────────────────────────────┐  │
│ │ 动作        组数  次数  重量    热身          │  │
│ ├───────────────────────────────────────────────┤  │
│ │ 杠铃卧推    4     10    60kg    ○            │  │
│ └───────────────────────────────────────────────┘  │
│                                                      │
│ [+ 添加动作]                                         │
│                                                      │
│           [取消]                      [保存记录]    │
└─────────────────────────────────────────────────────┘
```

**热身组自动判断**：
- 规则1：第一组 + 重量 < 历史最大重量50% → 热身组
- 规则2：第一组 + 次数 > 15 → 热身组
- 规则3：用户手动切换优先

**异常场景**：
- 解析失败 → 引导式提问 + 手动录入入口
- 部分解析 → 卡片展示 + 缺省字段高亮提示
- 动作模糊 → 候选选择卡片确认

**饮食记录对话流**
- 拍照识别食物 → AI估算营养成分 → 用户确认记录
- 文字描述食物 → AI解析 → 营养估算 → 确认记录

**问答对话流**（基于用户个人数据的个性化回答）
- 训练问题：动作选择、组数安排、高原期突破
- 营养问题：补剂用法、饮食搭配、热量计算
- 身体问题：肌肉酸痛、恢复建议、伤病初步判断

- 今日恢复评分（睡眠 + 上次训练强度）
- 肌群恢复热力图（哪些肌群尚未恢复，辅肌群同步纳入计算）
- 今日训练建议（练 / 休 / 轻度活动）

**容量加权恢复算法：**

每次训练记录保存时，将该组次的训练容量按肌群分配写入 `muscleVolumes`：
- 主肌群：容量 × 1.0（100% 归属）
- 辅肌群：容量 × 0.5（50% 归属）

例：杠铃卧推 60kg × 10次 = 600 容量
- 胸肌（主）→ 600
- 三头（辅）→ 300

恢复评分计算公式：
```
score = elapsedHours / (recoveryHours × volumeFactor) × 100
volumeFactor = clamp(actualVolume / baselineVolume, 0.5, 2.0)
```
- 低容量训练（< 基准）→ volumeFactor < 1 → 恢复更快
- 高容量训练（> 基准）→ volumeFactor > 1 → 恢复更慢

各肌群基准容量：腿部 7000kg/周，胸部/背部 5000kg/周，手臂 2500kg/周，核心 2000kg/周。

### Tab 4：我

#### 个人档案
- 基本信息（身高/目标体重/健身目标/当前阶段）

- 当前目标（目标体重 / 力量目标 / 时间节点）

#### 成就系统

**力量里程碑**
- 深蹲首次突破 60 / 100 / 140 / 180kg
- 卧推首次突破 60 / 100 / 120kg
- 硬拉首次突破 80 / 120 / 160kg
- 任意动作创造个人PR

**坚持里程碑**
- 累计训练 10 / 50 / 100 / 365 次
- 连续打卡 7 / 30 / 100 天
- 同一计划坚持超过 4 周

**身体里程碑**
- 体重变化达目标 25% / 50% / 100%
- 腰围减少 5cm / 10cm
- 连续30天记录体重

**记录习惯里程碑**
- 连续7天记录饮食
- 累计上传10张进度照片

**2. 补剂** ⭐（新增独立分类）

> 目标：帮助用户科学认知补剂、正确选择和使用补剂

**2.1 增肌支持类**

| 补剂 | 入门内容 | 进阶内容 |
|------|---------|---------|
| 蛋白粉 | 定义与类型（浓缩/分离/水解）、基本用法、训练后30分钟内冲泡 | 吸收速率差异、剂量计算（g/kg体重）、产品选择与性价比 |
| 肌酸 | 提升力量原理、每天5g维持期方案 | ATP-PCr机制、五天饱和法vs维持期、不同形式效果对比 |
| BCAA | 定义、亮氨酸/异亮氨酸/缬氨酸作用 | 与蛋白粉对比、边际效益分析、适用场景判断 |

**2.2 减脂辅助类**

| 补剂 | 入门内容 | 进阶内容 |
|------|---------|---------|
| 咖啡因 | 提升代谢原理、训练前30分钟、每日≤400mg安全剂量 | 个性化剂量计算、耐受性研究、与其他成分叠加效果 |
| 左旋肉碱 | 脂肪代谢作用、有氧前30-60分钟服用 | 必须配合运动才有效、内源性合成关系 |

**2.3 促进恢复类**

| 补剂 | 入门内容 | 进阶内容 |
|------|---------|---------|
| 谷氨酰胺 | 免疫系统支持作用、大强度训练后补充 | 肠道屏障维护、普通人是否需要补充的科学证据 |
| Omega-3鱼油 | EPA/DHA抗炎作用、每日1-3g建议量 | Omega-3/6比例、rTG vs EE vs TG形式吸收率、IFOS认证 |

**2.4 基础健康优化类**

| 补剂 | 入门内容 | 进阶内容 |
|------|---------|---------|
| 维生素D | 骨骼肌肉作用、每日1000-4000IU建议量 | 血液检测优化剂量、与K2协同作用 |
| 镁 | 肌肉收缩放松作用、睡前服用 | ATP合成关系、氧化镁/柠檬酸镁/甘氨酸镁吸收率对比 |
| 锌 | 睾酮合成、免疫功能、每日11mg建议量 | 与钙铁间隔服用、长期过量副作用 |

**2.5 训练周期专用类**

| 补剂 | 入门内容 | 进阶内容 |
|------|---------|---------|
| 氮泵 | 泵感原理（一氧化氮扩张血管）、训练前20-30分钟 | 成分表解读、咖啡因依赖性、有效性评估 |
| 恢复冲剂 | 训练后营养窗口期概念 | 蛋白质/碳水配比、与完整正餐时间安排 |
| β-丙氨酸 | 延缓疲劳机制、每日2-5g分次服用 | 2-4周见效周期、刺痛感无害、与肌酸叠加效果 |

**补剂分类内容难度标签**：`入门` / `进阶`

**AI主动推荐触发场景**：

| 用户行为 | 推荐内容 |
|---------|---------|
| 用户询问"该不该吃蛋白粉" | 补剂 → 增肌支持类 → 蛋白粉（入门） |
| 用户完成首次力量训练记录 | 补剂 → 增肌支持类 → 肌酸（入门） |
| 用户表示想减脂 | 补剂 → 减脂辅助类 → 咖啡因/左旋肉碱（入门） |
| 用户询问补剂是否有副作用 | 对应补剂子类的「常见误区」辟谣内容 |

**与现有内容的关系**：
- 原「身体营养 → 补剂指南」内容迁移至「补剂」各子类
- 原「AI辟谣 → 补剂误区」中的补剂相关内容整合至「补剂」各子类的「常见误区」板块

**4. 运动恢复**

> 目标：让用户重视恢复，把恢复当训练的一部分

| 子分类 | 内容主题 |
|--------|---------|
| 睡眠与恢复 | 睡眠质量对肌肉合成/皮质醇/训练表现的影响，优化睡眠的方法 |
| 延迟性肌肉酸痛（DOMS） | DOMS 的成因、何时需要担心、如何区分损伤与正常酸痛 |
| 超量恢复原理 | 训练→破坏→恢复→超量恢复的生理机制，如何利用训练周期 |
| 主动恢复方法 | 低强度有氧、拉伸、泡沫轴、游泳作为恢复手段的使用方法 |
| 被动恢复方法 | 冰浴/冷热交替浴、按摩、红外线疗法的科学依据与使用场景 |
| 减量周安排 | 何时安排减量周、减量周的训练量/强度如何调整 |
| 心率变异性（HRV） | HRV 反映的恢复状态解读，如何用 HRV 指导训练强度 |
| 过度训练综合征 | 判断过度训练的信号（力量下降/情绪低落/持续疲劳），应对方法 |

**5. 动作细节**

> 目标：帮用户掌握标准动作，避免错误代偿和受伤

每个动作条目包含：**目标肌群 → 标准动作要点 → 常见错误 → 变式选择 → 进阶/退阶方案**

## 四、AI激励机制

### 设计原则

- **文案不固定**：所有激励文案由 AI 实时生成，不使用硬编码模板
- **字数限制**：每条激励文案 ≤ 30 个汉字
- **数据感**：文案引用用户真实数据，让用户感受到 AI 在关注自己
- **语气差异化**：根据触发状态动态调整语气，避免千篇一律

### 核心算法

**e1RM（估算最大重量）计算**（Epley 公式）：
```
e1RM = 重量 × (1 + 次数 / 30)
示例：100kg × 8次 → e1RM ≈ 127kg
```

### 触发类型与上下文数据结构

每次触发激励时，系统组装上下文包传给 AI：

```json
{
  "trigger": "<触发类型>",
  "user": {
    "nickname": "用户昵称",
    "level": "beginner | intermediate",
    "streak": 14
  },
  "event": {
    "exercise": "动作名称",
    "prev_e1rm": 125,
    "new_e1rm": 131,
    "delta": 6,
    "unit": "kg"
  },
  "constraints": {
    "max_chars": 30,
    "tone": "<语气标签>",
    "no_repeat_last": ["最近3条激励文案内容"]  // 避免重复，取最近3条
  }
}
```

### 触发场景定义

| 触发类型 | 触发条件 | 传入核心数据 | 语气标签 |
|---------|---------|------------|---------|
| `pr_broken` | 本次 e1RM 超过历史最高 | 动作名、旧e1RM、新e1RM、提升幅度 | energetic |
| `pr_approaching` | 近3次 e1RM 持续上升 | 动作名、距PR差值、近期趋势 | anticipatory |
| `plan_completed` | 本周完成率 90%-110% | 完成率、连续达标周数 | affirming |
| `plan_behind` | 本周完成率 60%-90% | 缺少次数、剩余天数 | gentle |
| `plan_abandoned_risk` | 连续缺训 ≥ 3 天 | 缺训天数、原计划强度 | caring |
| `volume_milestone` | 本周容量较上周提升 ≥ 5% | 本周容量、提升百分比 | proud |
| `streak_milestone` | 连续打卡达 7 / 30 / 100 天 | 连续天数 | celebratory |
| `plateau_detected` | 同动作 e1RM 连续 2 周变化幅度 < 1% | 停滞动作、停滞周数、当前e1RM | analytical |
| `comeback` | 中断 ≥ 7 天后首次恢复训练 | 中断天数、上次训练内容 | warm |
| `goal_near` | 目标完成度 ≥ 80% | 目标内容、完成百分比、预计达成日 | countdown |

### AI 生成约束（系统 Prompt 核心片段）

```
你是健身助手"AI己"，根据用户训练数据生成一句激励文案。

规则：
- 不超过 30 个汉字
- 不使用感叹号堆砌
- 每次措辞不同（参考 no_repeat_last 避免重复）
- 根据 tone 标签调整语气
- 尽量引用 event 中的具体数字
- 只输出文案本身，不输出任何其他内容
```

### 执行时序

```
用户完成训练记录
       ↓
训练后分析引擎（并行执行）：
  1. 检测 PR 突破（比较 e1RM）
  2. 计算本次容量 vs 上次对比
  3. 更新本周计划完成率
  4. 更新连续打卡天数
  5. 检查里程碑触发
  6. 判断当前用户状态（进步期/平台期/疲劳期）
       ↓
组装上下文数据包 → 调用 AI 生成激励文案（≤30字）
       ↓
AI 对话流中：激励文案作为训练总结的第一句话
       ↓
如有 PR 突破：额外生成 PR 庆祝卡片（含分享按钮）
```

### 文案展示位置

| 位置 | 说明 |
|------|------|
| AI 对话（训练完成后） | 激励文案为 AI 回复第一句，后续跟进分析建议 |
| PR 庆祝卡片标题 | 简短有力的一句话 |
| 里程碑成就卡片 | 成就描述文案 |
| 微信推送通知正文 | 计划提醒/周报推送 |
| 周报开头 | 本周一句话总结 |
