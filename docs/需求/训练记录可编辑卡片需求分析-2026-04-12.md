# 训练记录可编辑卡片 需求分析

**日期**：2026-04-12
**版本**：v1.0
**状态**：已确认
**类型**：需求拓展分析

---

## 一、背景回顾

### 1.1 现有训练记录对话流

根据设计文档（docs/superpowers/specs/2026-04-12-aiji-miniprogram-design.md）：

**现有流程**：
```
用户描述训练 → AI解析 → 生成记录草稿 → 用户确认保存
```

**示例**：
- "我今天练了胸" → AI解析 → 生成记录草稿 → 用户确认保存
- "刚做了5组深蹲，120kg×5" → 自动识别动作 + 重量 + 次数 + 组数

### 1.2 用户需求

AI解析后生成**可编辑卡片**，用户可直接在卡片上修改：
- 动作名称
- 重量
- 次数
- 组数
- 热身组标记

---

## 二、核心问题分析

### Q1: 卡片的交互形式（如何展示可编辑字段）

**设计决策：内联编辑模式（Inline Editing）**

用户无需切换页面或打开模态框，直接在卡片内完成所有编辑操作。

**卡片结构设计**：

```
┌─────────────────────────────────────────────────────┐
│  💪 训练记录                                    [?] │
├─────────────────────────────────────────────────────┤
│                                                     │
│  📅 2026-04-12  胸部训练                           │
│                                                     │
│  ┌─────────────────────────────────────────────┐   │
│  │ 动作        组数  次数  重量    热身  操作   │   │
│  ├─────────────────────────────────────────────┤   │
│  │ 杠铃卧推    4     10    60kg    ○    [删除] │   │
│  │ 哑铃上斜    3     12    28kg×2  ○    [删除] │   │
│  │ 蝴蝶机      3     15    25kg×2 ○    [删除] │   │
│  └─────────────────────────────────────────────┘   │
│                                                     │
│  [+ 添加动作]                                      │
│                                                     │
│  ─────────────────────────────────────────────     │
│                                                     │
│  备注：状态不错，力量有提升                         │
│                                                     │
│           [取消]                    [保存记录]      │
└─────────────────────────────────────────────────────┘
```

**交互模式说明**：

| 交互形式 | 适用场景 | 用户操作 |
|---------|---------|---------|
| 点击编辑 | 动作名称 | 点击后变为下拉选择器 |
| 点击编辑 | 组数/次数 | 点击后变为数字输入框 |
| 点击编辑 | 重量 | 点击后变为数字输入框（带单位） |
| 点击切换 | 热身组 | 点击圆形按钮切换 ●/○ |
| 滑动删除 | 删除动作 | 左滑显示删除按钮 |
| 点击添加 | 新增动作 | 点击「+ 添加动作」展开动作选择器 |

**编辑状态视觉反馈**：

```
普通状态：
┌─────────────────────────────────┐
│  杠铃卧推    4     10    60kg   │
└─────────────────────────────────┘

编辑状态（点击任意字段）：
┌─────────────────────────────────┐
│  [▼ 杠铃卧推] [4] [10] [60kg]  │
│                                 │
│  ════════════════════════════  │  ← 蓝色边框表示可编辑
│  [完成]  [取消]                 │
└─────────────────────────────────┘
```

**移动端适配考虑**：
- 所有输入控件采用大触控区域（最小44×44px）
- 下拉选择器展开为半屏底部弹窗
- 数字输入使用微信数字键盘
- 重量单位（kg/lb）可通过点击切换

### Q2: 字段设计

**核心数据模型**：

```typescript
interface ExerciseSet {
  id: string;                    // 唯一标识（UUID）
  exerciseId: string;            // 动作ID（关联动作库）
  exerciseName: string;          // 动作名称（冗余存储，便于展示）
  sets: number;                   // 组数
  reps: number;                   // 次数
  weight: number;                // 重量
  weightUnit: 'kg' | 'lb';       // 重量单位
  isWarmup: boolean;             // 是否热身组
  notes?: string;                // 单组备注（如"力竭"、"辅助"）
}

interface TrainingRecord {
  id: string;
  date: string;                  // 日期 YYYY-MM-DD
  trainingType?: string;         // 训练类型（胸/背/腿等）
  exercises: ExerciseSet[];       // 动作列表
  totalVolume: number;           // 总容量（计算字段）
  notes?: string;                // 训练备注
  status: 'draft' | 'confirmed'; // 状态
  createdAt: Date;
  updatedAt: Date;
}
```

**字段定义**：

| 字段 | 类型 | 说明 | 编辑方式 |
|-----|------|------|---------|
| 动作名称 | string | 从动作库选择，支持搜索 | 下拉选择器 + 搜索 |
| 组数 | number | 1-20组 | 数字输入框，步进器 |
| 次数 | number | 1-100次 | 数字输入框 |
| 重量 | number | 支持小数 | 数字输入框 |
| 重量单位 | kg/lb | 用户偏好设置 | 点击切换 |
| 热身组 | boolean | 是否热身组 | 圆形切换按钮 |

**扩展字段（Phase 2考虑）**：

| 字段 | 类型 | 说明 |
|-----|------|------|
| RPE/自觉用力系数 | 1-10 | 用户选择 |
| 组间休息 | 秒 | 数字输入 |
| 完成状态 | boolean | 本组是否完成 |

### Q3: 热身组自动判断逻辑

**设计决策：AI智能判断 + 用户手动调整**

**自动判断规则**：

```typescript
function isWarmupSet(exerciseName: string, setNumber: number, weight: number, reps: number): boolean {
  // 规则1：如果是第一个动作的第一组，且重量较低 → 热身组
  // 规则2：重量明显低于该动作的历史最大重量（如 < 50%）→ 热身组
  // 规则3：次数明显多于常规次数（如 > 15次）→ 热身组
  // 规则4：用户手动标记优先
}

interface WarmupRule {
  condition: (ctx: WarmupContext) => boolean;
  priority: number;  // 优先级，数字越小越优先
}

const warmupRules: WarmupRule[] = [
  // 规则1：第一组 + 低重量
  {
    condition: (ctx) => ctx.setNumber === 1 && ctx.weight < ctx.historyMaxWeight * 0.5,
    priority: 1
  },
  // 规则2：第一组 + 高次数
  {
    condition: (ctx) => ctx.setNumber === 1 && ctx.reps > 15,
    priority: 2
  },
  // 规则3：显著低于历史重量
  {
    condition: (ctx) => ctx.weight < ctx.historyMaxWeight * 0.3,
    priority: 3
  }
];
```

**热身组标记的展示与交互**：

```
┌─────────────────────────────────────────────┐
│  动作        组数  次数  重量    热身        │
├─────────────────────────────────────────────┤
│  杠铃卧推    4     10    60kg    ○  ← 点击切换│
│  哑铃上斜    3     12    28kg×2  ●  ← 热身组 │
│  蝴蝶机      3     15    25kg×2  ○           │
└─────────────────────────────────────────────┘
```

**热身组视觉区分**：
- 热身组行显示为浅灰色背景
- 重量显示带有 "(热身)" 标签
- 热身组不计入总训练容量

**用户修正**：
- 用户可随时点击切换热身/正式组
- 系统记录用户修正行为，用于优化AI判断模型

### Q4: AI解析后的数据如何填充到卡片

**AI解析 → 卡片填充流程**：

```
用户输入：「今天练胸，杠铃卧推60公斤10次4组，哑铃上斜28公斤12次3组，蝴蝶机25公斤15次3组」
         ↓
┌─────────────────────────────────────────┐
│  AI解析引擎                              │
│  • 意图识别：记录训练                    │
│  • 实体提取：                             │
│    - 动作：卧推(胸)、上斜卧推(胸)、蝴蝶机 │
│    - 组数：4、3、3                        │
│    - 次数：10、12、15                     │
│    - 重量：60kg、28kg×2、25kg×2           │
│  • 训练类型推断：胸部训练                 │
└─────────────────────────────────────────┘
         ↓
┌─────────────────────────────────────────┐
│  生成草稿数据                            │
│  {                                       │
│    date: "2026-04-12",                  │
│    trainingType: "胸部训练",             │
│    exercises: [                         │
│      {动作Id, "杠铃卧推", sets:4, reps:10, weight:60, isWarmup: false},  │
│      {动作Id, "哑铃上斜卧推", sets:3, reps:12, weight:28, weightUnit:"kg×2", isWarmup: true},  │
│      {动作Id, "蝴蝶机夹胸", sets:3, reps:15, weight:25, weightUnit:"kg×2", isWarmup: false}   │
│    ]                                     │
│  }                                       │
└─────────────────────────────────────────┘
         ↓
┌─────────────────────────────────────────┐
│  渲染可编辑卡片                          │
│  （填充到对话卡片，详见Q1卡片结构）       │
└─────────────────────────────────────────┘
```

**AI解析增强策略**：

| 输入类型 | 解析策略 | 示例 |
|---------|---------|------|
| 完整描述 | 结构化提取所有字段 | "卧推60kg10次4组" → 动作+重量+次数+组数 |
| 部分描述 | 缺省字段使用默认值 | "练了胸" → 仅动作，需后续编辑 |
| 模糊输入 | 多候选+用户确认 | "做了几个动作" → 需要逐个确认 |
| 语音输入 | 同文本解析 | 微信语音转文字后解析 |

**动作识别模糊匹配**：

```
用户输入：「今天做了蝴蝶机夹胸」

AI解析候选：
┌─────────────────────────────────────────┐
│  找到以下相似动作，请确认：              │
│                                         │
│  ○ 蝴蝶机夹胸（器械）                   │
│  ○ 龙门架夹胸（绳索）                   │
│  ○ 哑铃飞鸟（自由重量）                 │
│                                         │
│  [确认]  [手动输入]                     │
└─────────────────────────────────────────┘
```

```
                    └──────┬──────┘

    │  用户点击   │
```

**用户操作细节**：

| 操作 | 触发 | 结果 |
|-----|------|------|
| 点击字段 | 编辑态切换 | 字段进入编辑模式 |

**Prisma Schema 扩展建议**：

```prisma
model TrainingRecord {
  id            String   @id @default(uuid())
  userId        String
  date          DateTime @db.Date
  trainingType  String?  // 如"胸部"、"背部"、"全身"
  notes         String?
  status        TrainingStatus @default(DRAFT)
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt

  user       User             @relation(fields: [userId], references: [id])
  exercises  TrainingExercise[]
}

model TrainingExercise {
  id                String   @id @default(uuid())
  trainingRecordId  String
  exerciseId        String
  exerciseName      String   // 冗余存储，便于展示
  sets              Int
  reps              Int
  weight            Float
  weightUnit        WeightUnit @default(KG)
  isWarmup          Boolean  @default(false)
  notes             String?

  trainingRecord TrainingRecord @relation(fields: [trainingRecordId], references: [id])
  exercise       Exercise       @relation(fields: [exerciseId], references: [id])
}

model Exercise {
  id          String   @id @default(uuid())
  name        String   @unique
  muscleGroup MuscleGroup
  equipment   Equipment
  isCompound  Boolean  @default(false)
  standardSets Int?
  standardReps Int?
}

enum TrainingStatus {
  DRAFT
  CONFIRMED
}

enum WeightUnit {
  KG
  LB
}

enum MuscleGroup {
  CHEST
  BACK
  SHOULDERS
  BICEPS
  TRICEPS
  QUADRICEPS
  HAMSTRINGS
  GLUTES
  CALVES
  ABS
  FULL_BODY
}
```

### Q7: 异常场景处理（AI解析失败/部分解析成功）

**异常场景分类**：

| 场景 | 描述 | AI响应策略 |
|-----|------|-----------|
| A. 完全解析失败 | 无法识别任何有效信息 | 引导式提问 + 手动录入入口 |
| B. 部分解析成功 | 仅识别部分字段（缺组数/重量等） | 卡片展示+缺省字段高亮提示用户补充 |
| C. 动作识别模糊 | 找到多个相似动作 | 候选选择卡片让用户确认 |
| D. 数据冲突 | 输入数据不合逻辑（如组数=0） | 提示修正，保留编辑能力 |
| E. 网络异常 | 解析服务不可用 | 本地缓存草稿 + 稍后重试 |

---

**场景A：完全解析失败**

```
用户输入：「今天练了一下」

AI响应：
┌─────────────────────────────────────────┐
│  我理解你想记录今天的训练～              │
│                                         │
│  可以告诉我具体做了哪些动作吗？           │
│  比如：卧推、深蹲、硬拉这些。            │
│                                         │
│  [帮我想想练什么]  [我忘了做了什么]       │
└─────────────────────────────────────────┘
```

**备选入口**：
- 「帮我想想练什么」→ 推荐训练快捷按钮
- 「我忘了做了什么」→ 进入模糊记录模式（仅记录训练日期和类型）

---

**场景B：部分解析成功**

```
用户输入：「做了卧推，60公斤」

AI响应：
┌─────────────────────────────────────────┐
│  找到以下记录，请确认：                   │
│                                         │
│  ┌───────────────────────────────────┐ │
│  │ 动作     组数  次数  重量   热身   │ │
│  ├───────────────────────────────────┤ │
│  │ [▼卧推]  [?]  [?]  60kg   ○      │ │
│  └───────────────────────────────────┘ │
│                                         │
│  ⚠️ 组数和次数还没告诉我，               │
│     请补充或确认默认值。                 │
│                                         │
│           [取消]           [确认保存]   │
└─────────────────────────────────────────┘
```

**缺省值规则**：

| 字段 | 缺省值 | 来源 |
|-----|-------|------|
| 组数 | 3 | 动作库 standardSets 或固定默认值 |
| 次数 | 10 | 动作库 standardReps 或固定默认值 |
| 重量 | 最近一次记录 | 训练历史 |
| 训练类型 | 根据动作推断 | 动作→肌群映射 |

---

**场景C：动作识别模糊**

```
用户输入：「今天做了夹胸」

AI响应：
┌─────────────────────────────────────────┐
```

## 六、优先级建议

### Phase 1（MVP）- 核心可编辑卡片

**核心功能**：
1. AI解析生成可编辑卡片
2. 基础字段编辑（动作/组数/次数/重量）
3. 热身组手动切换
4. 卡片确认/取消操作
5. 正常场景处理

**不纳入MVP**：
- 热身组自动判断
- 动作模糊匹配选择
- 本地草稿缓存
- 网络异常重试

### Phase 2（增强）

1. 热身组自动判断逻辑
2. 动作模糊匹配 + 候选选择
3. 添加动作的搜索功能
4. 滑动删除动作

### Phase 3（完善）

1. 本地草稿缓存
2. 网络异常重试机制
3. RPE/组间休息等扩展字段
4. 训练记录修改（编辑历史）

6. **保存后流程**：保存后是否自动进入「今日训练」执行界面？

## 八、附录：交互状态对照表

| 状态 | 触发条件 | 卡片展示 | 可用操作 |
